# SQC proof image cache — 2026-05-07

## Request
Andreas noticed the generated victory-scroll image felt like it was generated every time the page was visited and asked whether we could generate it once and save/reuse it for faster display.

## Reality
The proof image is deterministic and generated by `/api/og/proof/[token]`. We do not currently have persistent object storage wired for these images, so the fastest low-risk production fix is to make the deterministic route aggressively cacheable by browser/CDN.

## Change
- Added long-lived immutable cache headers to `/api/og/proof/[token]`:
  - `Cache-Control: public, max-age=31536000, s-maxage=31536000, immutable`
- Updated `ProofImage` to wait for browser timezone before setting the image `src`, so a page visit no longer asks for both:
  - `/api/og/proof/[token]`
  - `/api/og/proof/[token]?tz=...`
- Added a same-size placeholder while the client computes the timezone URL.

## Verification
- `pnpm lint` passed with warnings only.
- `pnpm build` passed.
- Production smoke should confirm cache headers on the proof image route after deploy.

## Future option
If we later want true stored files instead of route/CDN cache, add Vercel Blob/S3/R2 and persist the generated PNG at proof-completion time. That is heavier because it requires storage credentials, write path, cleanup policy, and migration/backfill decisions.
